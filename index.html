核心概念

什么是 Harness Engineering？

Harness Engineering 是设计、构建和运维围绕 AI 模型的基础设施的工程学科，确保 agent 在生产环境中可靠、一致、大规模运行。

🐎

缰绳隐喻

模型是马——强大但不可预测。Harness 是缰绳、马鞍、嚼子，引导这股力量做有用的工作，而不是让它到处乱跑。工程师是骑手，提供方向。

🧠

核心区分

Agent 是那个被训练出来的神经网络本身。工程师不是在"造 Agent"，而是在给 Agent 搭建运作环境。模型提供智能，Harness 提供双手、眼睛和工作空间。

# Fundamental equation of agentic systems
Agent    = Model                              # the trained neural network
Harness = Tools + Knowledge + Observation + Actions + Permissions

Framework vs Harness：本质区别

维度 Agent Framework Agent Harness
主要角色提供构建 agent 的库和抽象提供执行 agent 的运行时基础设施
类比蓝图 / 建筑材料工厂车间 / 操作系统
设计哲学高度模块化，用户自由组装强主张（opinionated），内置决策
状态管理通常无状态或轻量状态持久化状态，跨会话恢复
安全控制提供工具，不强制治理内置策略、工具访问中介、安全边界
代表产品LangChain, CrewAI, AutoGenClaude Agent SDK, Codex harness

💡 关键：Harness 通常使用 Framework。LangChain 自己的术语——LangChain = framework，LangGraph = runtime，DeepAgents = harness。

核心架构

Harness 的 7 大核心组件

一个生产级 agent harness 需要覆盖 7 个维度。每个维度都在解决模型本身无法解决的问题。

01
📋

Context Management

管理 agent 看到什么、记住什么。包括工作上下文（当前 turn）、会话状态（当前任务）、长期记忆（跨任务）和上下文压缩策略。

working-context session-state RAG compaction
02
🔧

Tool Orchestration

管理工具的访问和执行。包括受控访问（文件系统、API、shell、数据库）、参数验证、沙箱执行和输出净化。工具应设计为原子操作。

tool-use sandboxing MCP atomic-design
03
✅

Verification Loops

确保输出质量的护栏机制。结合确定性规则（linters、CI/CD）和 AI 驱动验证（agents 审查 agents）。将失败视为系统问题永久修复。

linting pre-commit agent-review error-recovery
04
💾

Memory Persistence

跨步骤和会话的状态持久化。包括进度追踪（JSON/文件）、会话恢复（中断后继续）和语义检索（RAG 模式）。解决"AI 失忆"问题。

progress.json session-recovery vector-store
05
📊

Observability

追踪每次工具调用、决策和 token 使用。聚类错误模式识别系统问题。监控成本，实现 KV-cache 优化（可降低 10x 成本）。

logging metrics cost-control LangSmith
06
🔒

Safety & HITL

人机交互控制和安全机制。操作分类为安全/中等/危险三级。关键操作需人类批准。提供受控升级机制。

risk-classification approval-gates human-in-loop
07
🏗️

Architectural Patterns — 三层循环模型

社区广泛采用的标准心智模型，将 harness 设计分为三个嵌套循环：

The Three-Loop Architecture
Outer Loop

Project Level — 项目级

捕获意图（规格、架构文档、知识库）+ 治理机制（防止代码库腐化）

Middle Loop

Orchestration Level — 功能级

每个 feature 需要完整计划（需求、设计、任务拆解），验证通过后再进入下一任务

Inner Loop

Task Level — 任务级

写代码 → 验证 → 错误反馈 → 重试。这个循环的结构决定了 agent 是输出可工作代码还是"自信的垃圾"

历史演进

从 Prompt 到 Harness

AI 工程实践的三个时代，每个时代都在解决上一个时代留下的问题。

2024
Prompt Engineering 时代
优化单次输入提示词。有效但局限——无法解决 agent 在多步骤任务中的可靠性问题。
2025 年中
Context Engineering 时代
Karpathy 等人推广，LangChain + Anthropic 正式定义。通过 RAG、MCP、记忆和动态注入管理 agent 看到什么。成为 Harness 的一个组件。
2025 年末
隐式实践阶段
AutoGPT、Codex 等系统中已存在 harness 组件，但尚无统一术语。Reddit r/AI_Agents 开始讨论 "agent harness" 概念（2025/12/20）。
2026 年 2 月
概念正式化
Mitchell Hashimoto（HashiCorp 联合创始人）博客文章 "Engineer the Harness" 正式命名。OpenAI 发布 "Harness engineering: leveraging Codex in an agent-first world"，披露百万行代码实验。
2026 年 3 月
系统化阶段（当前）
Martin Fowler 撰文讨论。清华+哈工大发表 arXiv:2603.25723 论文将其学术形式化。社区工具生态涌现（Lore、Desloppify 等）。Reddit/HN 热烈讨论，"2026 = Harness Engineering" 成为社区共识。
2026–2027
标准化预测
AaaS 平台、Semantic Graphs、模型商品化。Harness 成为主要工程产物，模型仅是其中的商品组件。

工业实践

主要机构的 Harness 实践

Anthropic、OpenAI、LangChain 等机构如何定义和实现 Harness 工程。

Anthropic

Claude Agent SDK

Anthropic 通过两个核心实践体现 Harness Engineering：

  • Effective Harnesses for Long-Running Agents（2025/11 工程博客）— Initializer/Coding-Agent 模式，使用 claude-progress.txt 在会话间交接
  • MCP (Model Context Protocol) — 连接 agent 到工具、数据源、工作流的标准协议，现已成为跨厂商开放标准
  • Claude Agent SDK（原 Claude Code SDK）— 高层 harness，自动处理工具编排和安全控制
OpenAI

Codex Harness + App Server

OpenAI 最完整地体现了 Harness Engineering 理念：

  • Codex Harness — 驱动所有 Codex 体验（web、CLI、IDE、macOS app）的核心逻辑和 agent loop
  • Codex App Server — 双向 JSON-RPC 1.0 API，三个核心原语：Item（原子单元）/ Turn（工作单元）/ Thread（持久化会话）
  • Harness 定义：将脚手架、反馈循环、文档和架构约束编码为机器可读的工件
LangChain

Framework → Runtime → Harness

LangChain 最清晰地区分了三个层次：

  • LangChain = Agent Framework（构建工具/蓝图）
  • LangGraph = Agent Runtime（执行运行时）
  • DeepAgents = Agent Harness（强主张、内置决策的完整系统）
  • "Agent = Model + Harness"是 LangChain 官方博客的核心方程
